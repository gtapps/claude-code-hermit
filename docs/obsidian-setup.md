# Hermit Cortex — Obsidian Setup

The Hermit Cortex is an optional Obsidian surface over your hermit's state. Six pages give you a live view into what the hermit is doing, where it struggles, how it's evolving, and how much it still needs you.

Your hermit works without it. Run `/claude-code-hermit:obsidian-setup` only when you want the Obsidian view.

## What you'll see

At session 1, Brain.md shows your live session and empty Dataview tables. By session 10, Cortex.md starts showing fragile zones. By session 20, Evolution.md tells a real story. The cortex gets more useful the longer your hermit runs.

---

## Quick Start

1. Open Obsidian → "Open folder as vault" → select the repo root
2. Add `.obsidian/` to `.gitignore`
3. Run `/claude-code-hermit:obsidian-setup` — this creates `obsidian/` and all cortex pages
4. Open `obsidian/Brain.md` first

That's it. The skill handles the rest.

### Vault path mode

The skill asks how Obsidian accesses `.claude-code-hermit/`:

- **Symlink (recommended):** Creates `hermit-state -> .claude-code-hermit`. Dataview queries use `hermit-state/sessions` etc. Simpler paths, no plugin needed.
- **Folder Bridge:** No symlink. Requires the [FolderBridge](https://github.com/tescolopio/Obsidian_FolderBridge) community plugin to make dot-directories visible. Dataview queries use `.claude-code-hermit/sessions` etc.

## The Six Pages

| Page                 | Role                                                                          |
| -------------------- | ----------------------------------------------------------------------------- |
| **Brain.md**         | Live session, fragile zones, what needs attention. Start here.                |
| **Cortex.md**        | Computed mindstate: uncertainty, stability, regressions, operator dependence. |
| **Evolution.md**     | Change over time: first vs latest, cost trends, autonomy trajectory.          |
| **System Health.md** | Operational state. Live embed of agent health summary.                        |
| **Connections.md**   | Relationship map: which sessions created which proposals.                     |
| **Cortex Portal.md** | Graph center: links all pages, recent sessions, active proposals.             |

### Brain.md — the first wow moment

The first page you open. The live session embed shows what's happening right now. Below it: where things keep breaking (last 30 days), what just changed, and accepted proposals waiting for implementation.

### Cortex.md — the mindstate

Brain.md is operational (what's happening now). Cortex.md is interpretive (what does the pattern of recent work mean). Shows: active uncertainty (accepted proposals), fragile zones (all-time), recently resolved problems, regressions (incomplete sessions in areas where proposals were resolved), and operator intervention week-by-week.

### Evolution.md — change over time

First session vs latest. Cost trajectory. Autonomy trajectory. Proposal resolution time. The page that shows the hermit is actually getting better.

### System Health.md — operational state

Leads with the `state-summary.md` embed. Agent alerts, pending micro-proposals, response rate, last reflection.

### Connections.md — the relationship map

Generated by `scripts/build-cortex.js`. Shows which session created which proposal, and where each proposal was accepted. Also shows artifacts linked to sessions, proposals, or standing alone as domain artifacts. Open **local graph** (right-click tab → Open local graph) for the visual.

### Cortex Portal.md — graph center

The graph staging hub. Links every cortex page with brief descriptions, recent sessions, active proposals, recent artifacts, and the latest weekly review. Open **graph view** (Ctrl+G / Cmd+G) or the local graph here for the best visual — cortex pages form a dense inner core, session and proposal nodes radiate outward.

## Artifacts

Skill output files (calendars, reports, analysis, templates) can be connected to the Obsidian graph so they don't float as disconnected nodes.

### cortex-manifest.json

Lives at `.claude-code-hermit/cortex-manifest.json`. Declares which paths `build-cortex.js` should scan for artifacts.

```json
{
  "version": 1,
  "artifact_paths": ["relatorios", "templates/captions", "calendario-*.md"]
}
```

- Paths are **project-relative** (not relative to `.claude-code-hermit/`)
- Directories are scanned recursively for `.md` files
- Glob patterns (e.g. `calendario-*.md`) match files at that path level
- Empty array or missing file = no artifact scanning (backward compatible)

Created automatically during `/obsidian-setup` (with heuristic path suggestions) or during `/hermit-evolve` (empty starter). Edit directly to add or remove paths.

### Artifact frontmatter

Files in declared paths appear in Connections.md based on their frontmatter:

```yaml
---
title: "Calendário Semanal de Conteúdo"
created: 2026-04-08T14:32:00+01:00
source: session
session: S-003
proposal: PROP-011
tags: [content, calendar]
---
```

| Field      | Required | Purpose                                                         |
| ---------- | -------- | --------------------------------------------------------------- |
| `title`    | Expected | Display name in Connections.md and Cortex Portal.md             |
| `created`  | Expected | Recency sorting in Recent Artifacts                             |
| `source`   | Optional | `session`, `interactive`, or `routine` (default: `interactive`) |
| `session`  | Optional | Links artifact to a session node in the graph                   |
| `proposal` | Optional | Links artifact to a proposal node in the graph                  |
| `tags`     | Optional | Enables Dataview grouping (same vocabulary as sessions)         |

Session linkage is optional. Artifacts created outside a formal hermit session (interactive, ad-hoc, routine) still appear in the graph — they just don't have session lineage.

### How artifacts appear in Connections.md

| Condition                   | Section                                       |
| --------------------------- | --------------------------------------------- |
| Has `session` field         | **Sessions → Artifacts**                      |
| Has `proposal` field        | **Proposals → Artifacts**                     |
| Has frontmatter but neither | **Domain Artifacts**                          |
| No frontmatter at all       | **Unlinked Files** (nudge to add frontmatter) |

## Graph View — honest positioning

The global graph is a demo hook and screenshot surface. It shows the relationship structure, with the portal at center. Visually compelling.

The global graph is not the daily workflow. The daily workflow is Brain.md → Cortex.md → weekly review → Evolution.md. The local graph on Cortex Portal.md is more useful than the global view for understanding specific relationships.

Wikilinks exist only in generated pages inside `obsidian/` — not in tracked files. Obsidian's native backlinks panel won't show references on individual session or proposal pages. The graph works because Connections.md and Cortex Portal.md link outward to tracked files. This is a one-directional index.

## Maintenance

Connections.md and Cortex Portal.md are regenerated nightly at 23:30 via the `cortex-refresh` routine (added automatically by `/obsidian-setup`). You can also run `/claude-code-hermit:cortex-refresh` manually at any time.

The other four pages (Brain, Cortex, Evolution, System Health) are static Dataview queries — they update live as Dataview reads the hermit state directory. No maintenance needed.

---

## Two Hard Rules

### 1. No duplicate truth

The **repo is the single source of truth**. Obsidian reads from it via live queries. If Obsidian and the repo disagree, the repo wins.

### 2. Canonical flows come from the plugin

Sessions: `session-start`, `session-close`. Proposals: `proposal-create`, `proposal-act`. Reflection: `reflect`. **Never** use Obsidian's Templater or "New note" to create session or proposal files — this bypasses lifecycle tracking.

## Suggested Layout

- **Left pane:** `tasks-snapshot.md` — what steps are planned, what's done
- **Right pane (pinned):** `Brain.md` or SHELL.md — what's happening, blockers, findings
- **Bottom:** `Cortex Portal.md` — entry point to everything else

Pin a pane: right-click tab → "Pin".

## Dashboard Queries

All session reports and proposals include YAML frontmatter. Dataview reads these fields directly.

### Session History

````markdown
```dataview
TABLE date, task, cost_usd AS "Cost", duration, status, tags
FROM "{{HERMIT_PATH}}/sessions"
WHERE id
SORT date DESC
```
````

### Proposal Pipeline

````markdown
```dataview
TABLE title, status, category, session, created
FROM "{{HERMIT_PATH}}/proposals"
WHERE id AND (status = "proposed" OR status = "accepted")
SORT created ASC
```
````

### All Proposals (including dismissed/resolved)

````markdown
```dataview
TABLE title, status, category, session, created
FROM "{{HERMIT_PATH}}/proposals"
WHERE id
SORT created DESC
```
````

### Cost Dashboard

````markdown
```dataview
TABLE date, cost_usd AS "Cost", duration, tags
FROM "{{HERMIT_PATH}}/sessions"
WHERE id AND date >= date(today) - dur(7 days)
SORT date DESC
```
````

### Agent Health

````markdown
```dataview
TABLE active_alerts, suppressed_alerts, micro_pending, micro_approval_rate AS "Micro Rate", response_rate, last_reflection
FROM "{{HERMIT_PATH}}/state"
WHERE file.name = "state-summary"
```
````

Replace `{{HERMIT_PATH}}` with `hermit-state` (symlink) or `.claude-code-hermit` (Folder Bridge).

## Advanced: Multi-Hermit Setup

If you run multiple hermit instances, you can create a cross-hermit observatory:

```
observatory/
  dev-hermit/        -> symlink to project-a/.claude-code-hermit/
  accountant-hermit/ -> symlink to project-b/.claude-code-hermit/
  dashboard.md       -> cross-hermit Dataview queries
```

Open the `observatory/` directory as an Obsidian vault. Dataview queries work across symlinked directories.

### Fleet Progress (from Task Snapshots)

````markdown
```dataview
TABLE progress, updated
FROM ""
WHERE file.name = "tasks-snapshot"
```
````

### Fleet Health

````markdown
```dataview
TABLE active_alerts, suppressed_alerts, micro_pending, micro_approval_rate AS "Micro Rate", response_rate
FROM ""
WHERE file.name = "state-summary"
```
````

One row per hermit. Shows which agent is noisy, which is healthy, which has low proposal engagement.

## What NOT to Do

- Don't create session/proposal files from Obsidian — use skills (`/session-start`, `/proposal-create`)
- Don't edit YAML frontmatter in session reports or proposals — use skills (`/proposal-act`, `/session-close`) instead
- Don't edit `tasks-snapshot.md` — auto-generated by the cost-tracker hook every turn
- Don't edit `Connections.md` or `Cortex Portal.md` — auto-generated by `build-cortex.js`
- Don't edit `config.json` in Obsidian — use `/hermit-settings`
- Don't use Obsidian Bases write-back to change proposal status columns — status changes must go through `/proposal-act` or lifecycle tracking breaks
- Don't use `[[wikilinks]]` or `%%comments%%` in tracked files — your hermit reads standard markdown only
- Don't install plugins that modify markdown on save (Linter, Auto Link Title) — creates phantom diffs and breaks frontmatter
- Don't rename/move tracked files — your hermit expects specific paths (`S-001-REPORT.md`, `PROP-001.md`)
