# Security

Safety model for autonomous hermit agents — deny patterns, defense in depth, and operational recommendations.

---

> **Important:** Deny patterns are blocklist-based string matches. They block known dangerous commands but cannot prevent all harmful invocations. The default `auto` mode reduces (but does not eliminate) the need for container isolation by requiring classifier review of each action (requires CC 2.1.150+ and Max/Teams/Enterprise plan — operators on Pro or Haiku will fall back to `acceptEdits`). Always use container isolation when running with `bypassPermissions`. See [Known Limitations](#known-limitations) below.

## Deny Patterns

Block tool invocations regardless of permission mode. Generated by `/claude-code-hermit:hatch` (step 9) or `/claude-code-hermit:docker-setup`. Canonical source: `state-templates/deny-patterns.json`.

### Default set (all deployments)

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(chmod 777*)",
      "Bash(*sudo *)",
      "Bash(*> /etc/*)",
      "Bash(curl * | bash*)",
      "Bash(wget * | bash*)",
      "Bash(env)",
      "Bash(printenv)",
      "Bash(cat .env*)",
      "Bash(cat */.env*)",
      "Bash(cat ~/.ssh/*)",
      "Bash(cat ~/.aws/*)",
      "Bash(*API_KEY*)",
      "Bash(*SECRET*)",
      "Bash(*TOKEN*)",
      "Edit(**/.claude-code-hermit/OPERATOR.md)",
      "Write(**/.claude-code-hermit/OPERATOR.md)"
    ]
  }
}
```

### Always-on additions (Docker / tmux)

```json
{
  "permissions": {
    "deny": [
      "Bash(ssh *)",
      "Bash(docker *)",
      "Bash(kubectl *)",
      "Bash(npm publish*)",
      "Bash(git push --force*)",
      "Bash(git push origin main*)",
      "Bash(git reset --hard*)",
      "Bash(*--no-verify*)"
    ]
  }
}
```

`/claude-code-hermit:docker-setup` applies both sets (default + always-on). `/claude-code-hermit:hatch` hardened mode applies both sets minus `docker/kubectl/ssh` (valid on host). Hatch minimal mode applies the default set only.

## Defense in Depth

| Layer               | Where           | Enforcement           |
| ------------------- | --------------- | --------------------- |
| Deny patterns       | `settings.json` | Mechanical*           |
| Bash sandbox        | `settings.json` (`sandbox.*`) | Kernel-enforced (`bwrap`); credential path denies + optional network isolation — auto-configured by `/hatch` |
| Agent-level rules   | `agents/*.md`   | Instruction-following |
| Hook enforcement    | `hooks.json`    | Mechanical            |
| Config isolation    | Named volume    | Mechanical            |
| Container isolation | Docker/VM       | Mechanical            |
| Container hardening | docker-compose.hermit.yml | Kernel-enforced (defense in depth — `no-new-privileges` is load-bearing; cap drops + PID cap are incremental) |
| Advanced hardening (opt-in) | docker-compose.security.yml | Kernel-enforced (LAN block + DNS policy, resource bounds, sysctls — opt-in via `/docker-security`; LAN containment is the load-bearing layer here) |
| OPERATOR.md         | OPERATOR.md     | Instruction-following |

*\* **Deny rule caveat:** There are reported bugs where `Edit`/`Write` deny rules in `settings.json` are not always enforced — Claude Code has been observed editing files that match a deny pattern. Deny rules are the correct approach and best available option, but can't be treated as airtight today. For OPERATOR.md specifically, Claude Code's built-in `.claude/` directory protection adds a second layer. Combined, OPERATOR.md modification is unlikely but not impossible until Anthropic fixes deny rule enforcement.*

## Unattended Asks

A channel-primary hermit (`always_on: true`, at least one reachable channel) has no attached interactive surface: nobody is watching the tmux pane. Anything that calls a synchronous ask primitive there — `AskUserQuestion`, a native permission prompt, a harness-rendered dialog — stalls the session invisibly: the pane blocks waiting for a keypress, and from the channel the hermit just goes quiet. The taxonomy for handling this, in order of preference:

1. **Pre-authorize** — grant the permission ahead of time (`permission_mode`, deny patterns) so the prompt never fires.
2. **Out-of-process** — resolve the decision outside the blocked turn entirely (the pause/resume keyword flow).
3. **Async approval** — bridge the question to the channel and let the answer arrive as a later message, so the turn can move on. This is the model for both binding instances below.
4. **Fail-loud** — when a prompt can't be redirected, notify the operator that something is stuck rather than silently hanging.
5. **Native bridge** — defer to a first-party mirror/answer path once Anthropic ships one, and shrink custom machinery accordingly.

Never bypass — no auto-answering a permission prompt, no sending Escape on the operator's behalf. Deciding what an unattended agent is allowed to do is always the operator's call, made in advance (config) or asynchronously (channel reply), never invented by the hermit in the moment.

Two binding instances exist today:

- **Hermit-owned skills (async approval, cooperative):** every core skill either carries the Step-0 channel-reply marker and routes its asks through the micro-proposal bridge, or contains no interactive ask at all — enforced by a static contract test (`tests/channel-ask-contract.test.ts`) that fails CI the moment a skill adds an unbridged ask.
- **Everything else (async approval + fail-loud, deterministic):** built-in flows, other installed plugins' skills, and spontaneous model-initiated questions are invisible to that static scan. A PreToolUse hook (`scripts/ask-gate.ts`, matcher `AskUserQuestion`) denies the call on a channel-primary session with a redirect reason that steers the model to the channel reply tool plus a durable `micro-proposals.json` entry — deny-steering behavior is probe-verified, the model does not retry the tool. The gate fires only on the **managed unattended session**, identified by the `HERMIT_MANAGED` marker `hermit-start` stamps into that session's tmux env-file (never `settings.local.json` or the docker-compose env block). This scopes the deny per-session rather than by the project-wide `always_on` config: a hand-launched maintenance `claude` in the same project — or a `docker exec` shell — lacks the marker, so its interactive AskUserQuestion flows (`/hermit-evolve`, `/hermit-settings`, `/docker-setup`) are never denied out from under a present operator. The gate leans **allow** on any ambiguity, because a false-allow is caught by the watchdog's fail-loud check below while a false-deny would silently break the operator's flow. Prompts that can't be redirected this way (native permission dialogs, harness-rendered prompts below the tool layer) are instead caught by the watchdog, which recognizes a stalled dialog on the captured pane, pushes one deduped operator notification per episode, and — critically — suppresses its own wedge nudges and re-arm keystrokes for that pane so it never auto-answers the pending prompt. A channel whose recent sends are failing (revoked token, etc.) is not treated as a valid redirect target either — tracked via advisory `state/channel-health.json` (written by `lib/channel-send`) — so the ask falls through to the native prompt and the watchdog backstop above instead of being denied into a channel that can't deliver. Opt out globally with `ask_gate: false`, or per-session by launching with `HERMIT_ASK_GATE=off`; the watchdog check is unconditional on any `watchdog.enabled` hermit.

**Revisit trigger:** if Anthropic ships a mirror or native permission-request bridge that lets the Claude app notify or answer a pending dialog on a headless session, both the gate's scope and the watchdog's pattern check should shrink accordingly — check current mirroring behavior before assuming this custom machinery is still the whole story.

## Auto-mode Classifier

Claude Code's `--permission-mode auto` (the hermit's default) routes tool calls through a classifier that blocks anything it judges irreversible, destructive, or aimed outside the trusted environment — a second gate that runs after the permissions system, before a tool call executes. Live tmux probes (CC 2.1.201, `--model sonnet` — **haiku has no auto mode and silently falls back to `acceptEdits`, so it can never exercise the classifier**) settled how a hermit's own sealed settings writes interact with it:

- **Tiers and resolution.** `autoMode.hard_deny` blocks unconditionally — the only way past it is an out-of-session executor (see below). `autoMode.soft_deny` (which includes the classifier's built-in `[Self-Modification]` rule — an agent widening its own `permissions.allow`) is cleared by either explicit intent in the operator's own live message, or a seeded `autoMode.allow` exception. The `ask_gate` tier above is a different mechanism entirely (a hermit-owned `PreToolUse` hook), not the classifier.
- **`autoMode` config is read only from `~/.claude/settings.json`, `.claude/settings.local.json`, managed settings, or `--settings`** — never from a committed project `.claude/settings.json`. A hermit's sealed autoMode entries (below) are always written to `settings.local.json` for this reason, regardless of `hatch_target`.
- **Auto mode suspends wildcarded-interpreter allow rules.** Narrow `permissions.allow` entries normally resolve before the classifier runs, but every `Bash(bun */scripts/*.ts*)` rule in the hermit's sealed allow-list (`apply-settings.ts` `allow` op) is wildcarded, so under auto mode all of them re-enter the classifier anyway. A hermit cannot rely on its own base permissions to pre-clear its maintenance scripts on this permission mode.
- **The seeded exception.** `hatch` (attended) seeds a single declarative `autoMode.allow` entry plus `autoMode.environment` context — sealed inside `apply-settings.ts`'s `automode-seed` op, not editable by callers — stating that the hermit's sealed `apply-settings.ts` invocations are pre-authorized, including when triggered by a plugin upgrade rather than a live operator message. Tone matters here: a probe session flagged an over-argued exception as "planted pre-authorization justification," so the entry is one plain declarative sentence, not persuasion.
- **A channel reply can never authorize a permission grant.** Two independent gates discount it: the model itself treats a `<channel>`-enveloped privileged instruction as a likely injection attempt before making any tool call, and even past that, the classifier's own soft-deny provenance test rejects anything not sourced from the operator's live message. So a channel-resolved decision may only flip hermit config (e.g. `artifacts.publish_authorized`) — the actual permission write always happens either attended (hatch/evolve, where the operator's presence satisfies the intent test) or from an out-of-session executor: `hermit-start`'s boot-time grant runs as a plain OS process before the Claude session starts, so it is never subject to the classifier at all.
- **`PermissionDenied` notifier.** A `PreToolUse` gate can preempt a stall (see `ask_gate` above), but the classifier's own denials happen inside a tool call the hermit didn't anticipate. The `PermissionDenied` hook (`permission-denied-notify.ts`) makes those visible on the managed unattended session — a deduped, bounded channel message naming the blocked tool and its input (Claude Code's `PermissionDenied` payload delivers `tool_name`/`tool_input`, not the classifier's reason text — that stays in the session transcript and the Recently-denied view, so the alert points the operator at *what* was blocked, not the model's phrasing of *why*) — the deterministic counterpart to `CLAUDE-APPEND.md`'s "Auto-mode denial alert" rule. It cannot block or retry the call; auto-retrying a classifier denial would itself be the bypass this section describes.

## Bash Sandbox

`/hatch` auto-configures Claude Code's native bash sandbox (v2.1.150+) on probe pass. The standard profile:

- **Filesystem:** denies reads of `~/.aws`, `~/.ssh`, `~/.gnupg`
- **Network:** unrestricted (no outbound block at this layer — use `/docker-security` for egress control)

The sandbox wraps every `Bash` tool call with `bwrap` (Linux) or the platform equivalent. If the probe fails (bubblewrap unavailable), `/hatch` prints an install hint and skips.

**Docker:** for container deployments the bash sandbox is set off (`sandbox.enabled:false`) at `/hatch` / `/docker-setup` time. The container boundary (`cap_drop:ALL`, `no-new-privileges`, network policy) is the isolation layer Anthropic recommends for unattended runs, and the nested bwrap sandbox is optional defense-in-depth rather than required (https://code.claude.com/docs/en/sandbox-environments). It is left off for two reasons: on Ubuntu 24.04+ hosts (`kernel.apparmor_restrict_unprivileged_userns=1`, the 23.10+ default) the AppArmor `docker-default` policy blocks `bwrap` from creating user-namespaces in-container, so it can't start and would silently kill heartbeat and `/watch` Monitor subprocesses; and running it where it *can* start (a host without the userns block) needs `enableWeakerNestedSandbox` — which only addresses the in-container `/proc` case, not a blocked user-namespace — and which Anthropic notes "considerably weakens security." `hermit-start` does not manage this key at runtime — existing installs are migrated once by `/hermit-evolve` (which asks before changing it).

**Opting out:** set `sandbox.enabled: false` in your settings file.

**Re-enabling on a host (tmux):** if a `warn` probe disabled the sandbox at hatch time, install the bwrap AppArmor profile on the host (per the [sandbox docs](https://code.claude.com/docs/en/sandboxing)), then set `sandbox.enabled:true` and delete `.claude-code-hermit/state/sandbox-probe.json` to clear the cached probe result.

**Custom CA / MITM proxies:** tools that use a corporate proxy with a custom certificate authority may need `"enableWeakerNetworkIsolation": true` in the `sandbox` block. See the [FAQ](faq.md) for details.

## Container Hardening

`/claude-code-hermit:docker-setup` (v1.0.26+) renders three kernel-enforced hardening stanzas into `docker-compose.hermit.yml`. These are defense in depth — they reduce the blast radius of a successful prompt injection or a compromised plugin. They are **not** a substitute for the upstream defenses (deny patterns, non-root user, named-volume isolation, no host secret mounts) and they don't make the container "safe against prompt injection." They make a successful injection less useful.

Honest weighting:

| Stanza                                   | What it does                                                                   | Real teeth?                                                                                                 |
| ---------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| `security_opt: [no-new-privileges:true]` | Blocks setuid escalation at the kernel level, even via planted setuid binaries | **Yes** — closes a real vector against future supply-chain compromise of any installed plugin              |
| `cap_drop: ALL`                          | Revokes Linux capabilities (NET_RAW, SETUID, DAC_OVERRIDE, …)                  | Incremental — non-root `claude` user already had no ambient caps; this closes the kernel-enforced ceiling  |
| `pids_limit: 2048`                       | Caps process count (fork-bomb / runaway-subprocess bound)                      | Resource bound, not a security primitive on its own — limits collateral damage                             |

Hermit's runtime (tmux, claude CLI, bun, jq, npm, git+HTTPS plugin installs) needs none of what these stanzas remove. Steady-state PID usage is ~80; install bursts can spike higher but stay well below 2048.

**Override path.** Operators with custom workloads (services on ports <1024, setuid helpers, high-PID frameworks) edit `docker-compose.hermit.yml` directly to relax or remove the relevant stanza. The defaults are conservative because the recommended-plugins flow accepts third-party marketplaces — every operator who didn't audit every plugin benefits from the floor.

**Verifying it's active.** From the host:

```bash
# Capabilities — both should be 0000000000000000
docker compose -f docker-compose.hermit.yml exec hermit \
  grep -E '^Cap(Eff|Bnd)' /proc/self/status

# no-new-privileges — should report NoNewPrivs: 1
docker compose -f docker-compose.hermit.yml exec hermit \
  grep NoNewPrivs /proc/self/status

# PID limit — should report 2048
docker compose -f docker-compose.hermit.yml exec hermit cat /sys/fs/cgroup/pids.max
```

**Limitation.** These flags only apply at container creation, not on `restart`. After patching an existing compose file, run `hermit-docker down` then `hermit-docker up` — `hermit-docker restart` silently keeps the old container.

### Advanced hardening — `/docker-security`

Beyond the always-on baseline above, `/claude-code-hermit:docker-security` is an opt-in advanced wizard that adds a `docker-compose.security.yml` overlay with LAN containment + DNS policy (firewall + DNS sidecar with port-53 redirect for actual enforcement), resource bounds + kernel hygiene sysctls, and a boot-time plugin-install audit log. Each toggle is presented with honest cost/benefit framing, applied as an overlay (does not modify `docker-compose.hermit.yml`), verified against the live container, and fully reversible. The LAN containment toggle is hard-skipped when `docker.network_mode: "host"` is in use. Honest limitation surfaced in the wizard and dedicated docs: DNS policy blocks domain-based exfil but cannot stop direct-IP egress to a hardcoded public address. See [Docker Security](docker-security.md) for the full toggle reference, verify commands, and documented limitations.

## Recommendations

- **Strict hook profile** for always-on agents (no performance penalty)
- **Session budget** of $5-10 for overnight sessions
- **Heartbeat** for 24/7 monitoring — detects stalled agents, changed conditions, rate limits
- **Network:** allow only `api.anthropic.com`, your channel API, and `github.com` when possible
- **Secrets:** don't mount `~/.aws/`, `~/.ssh/` into containers. Use scoped API keys.
- **Review session reports** before pushing — check for leaked paths, credentials, or connection strings

### Known limitations

- **`hermit-doctor`'s `channel-liveness` check and the deterministic channel-send path (`lib/channel-send.ts`) are two of core's direct outward egress paths.** Doctor's liveness call (Telegram `getMe`, Discord `/users/@me`) sends no message content, 5s timeout, fail-soft. `channel-send.ts` does send message content — it is the mechanism behind the watchdog's lifecycle pushes (restart/wedge/forced-pause), `cost-tracker`'s budget-cap alerts, and the `channel-status-responder.ts` reply — reading the bot token straight from the already-configured channel's `state_dir/.env` (the same token the channel MCP plugin uses), a plain-text POST with an explicit 10s timeout, and no retry on failure (fail-open: the caller falls back to its existing model-mediated path). Every other doctor check is a local filesystem read. The docker-security DNS allowlist already covers both platform hosts (`telegram.org`, `discord.com`) — no allowlist change needed. Disabling a channel disables both its liveness probe and any push through it; there is no separate per-check opt-out in v1.
- **Remote control and status disclosure trust the operator DM when no `allowed_users` is set.** A plain channel *reply* still follows the documented accept-all default when `channels.<source>.allowed_users` is absent — but the two paths that either *mutate state* (`pause`/`resume`/`snooze` via `pause-keyword.ts`) or *disclose operator state* (the deterministic `status` reply) do **not**. With no allowlist configured they trust only the operator's own DM (`chat_id === channels.<source>.dm_channel_id`); any other sender's control keyword is a silent no-op, and any other accepted sender gets a redacted, coarse status (no spend figures, task text, or pending-approval IDs). **Caveat:** if a channel's primary chat *is* a group/server channel, its `dm_channel_id` equals that shared chat, so every member matches the DM binding — those installs must set `allowed_users` explicitly to restrict control and full status to named operators. This is a flat trust list, not RBAC (an RBAC pass on `allowed_users` is a deliberately-deferred enterprise gap).
- **Telemetry export is another direct outward egress — opt-in, off by default.** `config.telemetry_export` (absent or `enabled: false` = nothing ever leaves the box) periodically POSTs a versioned health/cost bundle to an operator-configured webhook from the watchdog tick. The bundle is allowlist-built: only named numerics/enums/counts are copied in by default (cost totals, doctor check statuses, alert counts, session cost/duration, watchdog event counts); free text (task descriptions, error messages, alert detail strings) is included only when the operator explicitly sets `redact_operator_text: false`. The bearer token comes from an env var named in config, never from `config.json` itself, and neither the webhook URL nor the token ever appear in logs or error output. Under docker-security, the webhook's host must be added to the DNS allowlist or exports fail (and spool) inside the hardened container.
- **Three artifact pages publish session-derived content to claude.ai — default on, per-page opt-out.** `config.artifacts.dashboard`/`proposals`/`weekly_review` (default `true`) each publish a rendered page — status/latest-brief/proposals/weekly-evolution, open-proposal full text, and the compiled weekly review, respectively — via Claude Code's built-in Artifacts feature, not a hermit-authored HTTP call. Same publish path as any other Artifact a session creates, so it's governed by Claude Code's own controls (`disableArtifact`, `CLAUDE_CODE_DISABLE_ARTIFACT`, `permissions.deny`), not the docker-security DNS allowlist. Private to the publishing account on Pro/Max; Team/Enterprise can additionally Share within the org, never across it. Every automatic publish is a deterministic render of enumerated state files — with one caveat: the dashboard's latest-brief section embeds `state/last-brief.json`, whose `text` field is itself model-composed by the `brief` skill, so that one section is not exclusively machine-generated content. On-demand publish of a compiled doc or proposal (operator-initiated, no config gate) is likewise a direct file publish, no model authorship. See `docs/config-reference.md#artifacts` and `docs/artifacts.md`.
- **`permissions.allow: ["Artifact"]` is a sanctioned pre-approval of an outward-publishing tool for autonomous use.** Applied via the `artifacts.publish_authorized` config flag (§ Auto-mode Classifier above) so unattended sessions don't stall on the first-publish permission ask (a headless "ask" is an effective deny) — a channel reply sets the flag only, and `hermit-start`'s boot-time grant, running outside any session, is what actually writes the permission (needed because the classifier blocks a session from widening its own `permissions.allow`, even attended-seeded). Re-ensured every boot, so a hand-wiped entry heals itself; revoke via `/hermit-settings artifact-authorization`. Assessment: on Pro/Max, artifacts are private to the publishing account with no share option, so a prompt-injected publish is persistence/noise, not exfiltration — the attacker cannot view the page, and `channel-send.ts` (above) remains the materially riskier egress. The real cost is losing the human checkpoint before irreversible external persistence: there is no tool-side delete, and published content may be cached even after removal from the gallery, so a mistaken publish (e.g. a render that embeds a secret that leaked into state) can't be fully recalled. Mitigation: every automatic-path publish is a deterministic render of enumerated state files (no free-form model-authored pages on the automatic path — see the brief-text caveat above); operators who decline the flag get the same outcome via setup-time banking (first publish happens inline, attended, at hatch/evolve time) instead of a standing grant.
- **Egress filtering is opt-in.** Bridge networking isolates from localhost but the container can still reach the internet by default. A successful prompt injection can exfiltrate data to external hosts. **Mitigation available**: `/claude-code-hermit:docker-security` adds LAN containment + DNS policy via a firewall sidecar with kernel-level port-53 redirect for real enforcement. **Remaining gap**: direct public-IP egress (hardcoded IPs, no DNS) is not blocked even with the wizard — v1.1 may add nftset-driven IP allowlisting. See [Docker Security](docker-security.md).
- **No input sanitization.** Content fetched from external APIs enters Claude's context unsanitized. Deny patterns cannot catch injection via fetched content — that's an instruction-following boundary, not a mechanical one. Future: pre-processing hook that strips known injection patterns.
- **Deny patterns are a blocklist.** They catch commands you anticipated. Equivalent commands not on the list execute normally. Bind mount scope and network isolation are stronger defenses for novel attacks.
- **Episodic channel-log (PROP-010) stores raw DM text locally.** `state/channel-log.sqlite` holds the operator's Discord/Telegram DM text verbatim — capture is gated by the same `channels.<source>.allowed_users` allowlist channel-responder enforces (unauthorized senders are never logged), but authorized senders' full message text is retained until weekly-review consolidation and its retention window prune it. Storage is deliberately unsanitized (fidelity for consolidation), but the `/recall` read path (`search.ts`) runs recalled text through `safeForLLM` — the same control-char/tag-impersonation defusal `channel-reply-reminder.ts` already applies to the envelope's `source`/`chat_id` — before it reaches model context. Treat it as untrusted external input regardless: it's the operator's own words, but unreviewed. The file is local-only, covered by the standard `state/` gitignore, and never committed. Disable capture with `knowledge.channel_log_enabled: false` if this tradeoff isn't wanted.

## Security Checklist

- [ ] Bash sandbox enabled — auto-configured by `/hatch` on supported hosts (check `sandbox.enabled` in settings file); `bubblewrap` required on Linux
- [ ] Docker/VM if using `permission_mode: "bypassPermissions"` — see [Always-On Setup](always-on.md)
- [ ] Non-root user inside container
- [ ] Container hardening stanzas present (`no-new-privileges`, `cap_drop`, `pids_limit`) — generated by `/docker-setup` v1.0.26+
- [ ] Advanced hardening considered (`/docker-security`) — at minimum LAN containment if hermit runs on a home/office network alongside HA / NAS / personal devices
- [ ] Deny patterns configured
- [ ] Config isolation via named volume (default) — no host `~/.claude` bind-mount
- [ ] No host mounts beyond project directory
- [ ] No production credentials accessible
- [ ] Strict hook profile
- [ ] Session budget set
- [ ] Heartbeat enabled
- [ ] OPERATOR.md includes approval constraints and hard rules
- [ ] Session reports reviewed before pushing
- [ ] `state/` directory gitignored (contains runtime observations, not config; also holds `channel-log.sqlite` if the episodic channel-log tier is enabled)
- [ ] Only trusted marketplaces in `docker.recommended_plugins` (review entries periodically via `/hermit-settings docker`)

## Plugin Security

### Trust Model

Installing a plugin runs that plugin's hooks and skills with the same authority as hermit. Only install plugins from sources you trust the same as hermit itself. The container hardening in `docker-compose.hermit.yml` (`cap_drop`, `no-new-privileges`, `pids_limit`) reduces what a *compromised* plugin can do to the host kernel — it does not vet the plugin's intent, sandbox its file access within the project, or prevent it from acting on your behalf via the agent. A malicious or careless plugin still runs as you.

Plugins run with the same permissions as Hermit. The default `auto` mode applies classifier review to each action; if the operator has opted into `bypassPermissions` for fully unattended Docker operation, plugins run with full unrestricted execution — no permission prompts, no human review.

**Policy:** The entrypoint installs every enabled entry in `docker.recommended_plugins` on first boot, regardless of marketplace. The trust gate is at **configuration time**: entries only land in `config.json` when the operator explicitly confirmed the mirrored plugin list during `/docker-setup` or `/hermit-settings docker`. Preselection during setup is restricted to `claude-plugins-official` and `gtapps/*` — third-party entries must be manually opted in. See [Recommended Plugins — Trust model](recommended-plugins.md#trust-model).

**No auto-updates:** The entrypoint installs plugins on first boot but never auto-updates them. Plugin updates are operator-initiated via `/claude-code-hermit:hermit-evolve` or image rebuild.

**Hermit does not vet, audit, or take responsibility for any plugin.** Even official plugins should be understood before enabling. You are responsible for evaluating what you install.

See [Recommended Plugins](recommended-plugins.md) for the full list and installation instructions.
